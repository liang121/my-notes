# Agent 如何避免无限递归

> Sources: 飞书 Wiki（研究 ROMA 框架的运行机制）, 2026-05-08
> Raw: [Agent 如何避免无限递归](../../raw/ai-agents/2026-05-08-agent-avoid-infinite-recursion.md)

## Overview

Agent 跑着跑着不停下来，本质是**它不知道"什么时候算完"**。打个比方：你让一个实习生"把这份报告改到满意为止"——没说清楚什么叫"满意"、最多改几遍、改不动了怎么办，他就可能一直改下去。预防 Agent 无限递归同理，要从**目标定清楚、强制设上限、状态认得准、外部别拖死、出事能查到**这五件事入手。下文把原文 17 条策略归成"为什么会卡死"和"怎么救"两块，每条都配一个具体场景。

## 为什么 Agent 会陷入无限递归

四类根因。一旦命中，就是"实习生不知道什么时候停"的具体表现：

**1. 目标模糊或循环依赖**
Agent 不知道"做完"长什么样，或者两个任务互相等。

> 例：Agent 要"验证 A"，但代码写成"验证 A 前先验证 B"，而验证 B 又调"验证 A"——栈炸为止。
> 现实里更常见的版本：`research(topic) → 调 critique() → critique 不满意又调 research()`，没有"够好就停"的标准。

**2. 触发条件没边界**
"结果不达标就重试"——但"达标"是什么没量化，"重试上限"也没设。

> 例：让 Agent 写文案"直到用户满意"。没人定义满意，于是它一稿、二稿、二十稿。

**3. 状态判断错了**
Agent 自己都搞不清当前是"在跑"还是"跑完了"，于是一遍遍重启自己。

> 例：用 `if status != "done": run_again()`，但 `status` 在并发场景下还没刷新就被读到，永远不是 `done`。

**4. 外部依赖卡住但没异常处理**
依赖的接口/工具一直返回"处理中"或不响应，Agent 没设超时，傻等到天荒地老。

> 例：调用一个第三方搜索 API，对方挂了，返回 `"pending"`。Agent 每秒轮询一次——一年后还在轮。

## 怎么救：五类策略 + 17 条具体做法

### 一、把目标和边界讲清楚（破除逻辑闭环）

**1. 目标拆解 + 量化输出**
把"完成订单"拆成可验证的子步骤，每一步的产出就是下一步的输入：

```
接收订单 → 校验合法性 → 扣库存 → 生成物流单 → 通知用户
```

每个箭头都有明确的"输出长什么样"。不再有"反复改到满意"这种含糊的回路。

**2. 禁止 A↔B 循环依赖**
设计期就用流程图过一遍依赖。如果 A 真的需要 B 的结果，加一个**中间判定节点**："B 结果满足条件吗？不满足就直接结束，而不是回头让 A 再试。"

> Agent 编排里很常见的反模式：planner → executor → critic → planner → ……
> 修法：critic 给出评分后，由**外层调度器**（不是 critic 自己）决定要不要再让 planner 跑，且最多 N 次。

### 二、设硬性上限（递归次数 + 执行时长）

**3. 最大递归次数（必做）**
每个能自调用/重试的模块都要有计数器：

```python
MAX_RETRY = 3
for i in range(MAX_RETRY):
    result = fetch_external_data()
    if result.ok:
        break
else:
    raise DataFetchFailed("3 次都失败，放弃并通知用户")
```

阈值要根据场景调，太小会误伤正常流程。

**4. 总执行时长上限（兜底）**
单个任务/整体任务都设 deadline。比如"用户查询"最长 10 秒，超时就退出 + 返回"超时请重试"。

> 实操：很多 Agent 框架（LangGraph、AutoGen 等）都有 `recursion_limit` 或 `max_iterations` 参数——**默认值不是 0**，意味着官方也认为这是必装的保险丝。

### 三、状态认得准（精准感知终止）

**5. 状态集合写死**
明确列出所有可能状态：`init / running / waiting_external / success / failed / terminated`，每个状态都有唯一判定标准。不要出现"看上去像 success 但又像 running"的暧昧地带。

**6. 多维度校验"成功"**
单一信号容易误判。例：判断"订单处理成功"不能只看"物流单生成成功"，要同时满足 `库存扣减 OK & 支付确认 OK & 物流单 OK`，三者全 OK 才叫 success。

> 反例：只看"代码已生成"就判定 Agent 完成 coding 任务——结果生成的是空文件，于是无限循环让它"重新完善"。改成 `文件非空 & 通过 lint & 测试通过` 三个条件都满足才停。

**7. 主动捕获异常状态，别傻等**
预先列出所有可能异常（接口无响应、数据格式错、token 超限……），每种都配一条专属处理路径：

```
检测到 "外部接口无响应"
  → 切换备用接口 / 终止流程并反馈
  ✗ 不要：继续每秒发请求等它好起来
```

### 四、稳住外部依赖（别让它把 Agent 拖死）

**8. 调用前先做可用性检测**
重要操作前先发一次轻量心跳/测试请求。例：调支付接口前先 ping 一下，挂了直接终止，别让真实订单卡在那里。

**9. 反馈格式要标准化**
和外部模块约定清晰的状态码：`200 成功 / 400 参数错 / 500 服务器异常`。Agent 看到 500 就知道"这条路走不通，走异常分支"，而不是看到一个含糊的 `"处理中"` 就一直等。

**10. 超时按异常处理**
没在预设时间内返回？默认判超时，跑超时分支（重试一次或直接告诉用户失败），而不是把超时当"还在处理，再等等"。

> 在 LLM 调用里特别重要：模型偶尔会"卡住"——不是返回错误，而是迟迟不出 token。给每次调用设 hard timeout 比依赖网络层超时更可靠。

### 五、监控 + 回溯（出了事能找到根）

**11. 实时监控**
搭一个简单 dashboard，盯每个模块的调用次数和耗时。某模块调用次数逼近上限、或耗时异常拉长，自动报警——这是"还没死循环但快了"的早期信号。

**12. 详细日志**
每次操作记录：时间、调用了哪个模块、输入输出、判定结果。出问题时能往回查："是哪一步的状态判定写错了？哪个阈值定低了？"

> 对 LLM Agent 尤其关键：因为 prompt 是动态的，没有完整的 trace 根本复现不出 bug。LangSmith / Helicone / 自建 logging 三选一。

**13. 上线前压测 + 边界测试**
故意制造极端情况：模拟外部接口持续返回脏数据、模拟目标描述故意写得很模糊、模拟阈值踩在临界点——看保险丝有没有按预期烧断。

## 一句话总结

> **Agent 不是不会停——是没人告诉它什么时候停。**

四个救命问题，每写一个 Agent 都过一遍：

1. "做完"长什么样？（量化的成功条件）
2. 最多跑几次/多久？（次数上限 + 时长上限）
3. 出错了走哪条分支？（异常状态预案）
4. 卡住了我能看见吗？（监控 + 日志）

四个都答得上来，就基本不会无限递归。
